# v1.4 UI refresh — design

Date: 2026-07-10
Status: approved (brainstormed with visual companion; user picked battery
option B, navbar option B, desktop bookmark behavior A)

Eight changes bundled for the next minor release (v1.4.0), grouped into six
work areas.

## 1. Chart interactions: factorize the shared config

The Home `TemperatureChart` already has wheel zoom, drag pan, pinch and
double-click reset — it shares `ChartCanvas` (`crates/chaos-ui/src/echarts.rs`)
with the weather charts. What is duplicated is the `dataZoom` "inside"
interaction fragment, copy-pasted in `pages/weather.rs` (twice) and
`pages/home.rs`.

- Add a helper in `echarts.rs` (e.g. `pub fn inside_zoom() -> serde_json::Value`)
  returning the shared fragment:
  `{"type": "inside", "zoomOnMouseWheel": true, "moveOnMouseMove": true,
    "moveOnMouseWheel": false}`.
- Replace the three inline copies with the helper. Existing tests asserting
  the flags move to `echarts.rs` (or keep asserting through the page option
  builders — implementer's choice, no behavior change).
- No new gestures: if pinch feels broken on Android today, the cause is the
  page itself being pinch-zoomable (fixed by §6). Verify chart pinch on an
  Android build after §6 lands.

## 2. Battery levels for home sensors

### Server (`crates/chaos-server/src/home_assistant.rs`, config)

- Battery entity is **auto-derived** from each configured sensor's
  `entity_id` by suffix swap:
  - `..._temperature` → `..._battery`
  - `..._temperature_N` → `..._battery_N` (N = digits, HA's dedup suffix)
  - If the entity id matches neither pattern, no derivation is attempted.
- New optional per-sensor config field `battery_entity_id` overrides the
  derivation (also the escape hatch for unmatchable names).
- When serving the home payload, fetch the battery entity's current state
  alongside temperatures. A missing entity, `unavailable`/`unknown` state, or
  unparsable value yields `None` — never an error and never blocks the
  temperature data.
- Domain: the per-sensor type in `chaos-domain/src/home.rs`
  (`HomeSensorInfo`, served by `/home/sensors`) gains `battery_pct: Option<f64>`
  (`skip_serializing_if = "Option::is_none"` to stay wire-compatible).

### UI (`crates/chaos-ui/src/pages/home.rs`)

- New **"Sensors" card** on the Home tab: one row per configured sensor —
  label, horizontal battery bar, percentage text.
  - Bar fill green normally, red below 20%.
  - Sensors with `battery_pct = None` show an em dash instead of the bar.
- Card uses existing card styling; rows follow the light-card row pattern.

### Tests

- Unit tests for the suffix-swap derivation (plain, `_N`, non-matching).
- Stub-HA server test: battery state fetched and surfaced; missing battery
  entity yields `None` while temperatures still flow.

## 3. Icon fallback tile

There is no llama.cpp icon in any proxied source (dashboard-icons, selfh.st,
simple-icons all lack it), so a blank square is inevitable server-side.

- UI fix, applied wherever icons render (services grid
  `components.rs`, bookmarks `pages/dashboard.rs`): on `<img>` error, replace
  with a **letter tile** — rounded square, first letter (uppercased) of the
  service/bookmark title, background color picked deterministically from a
  small palette by hashing the title.
- Implementation: an `on:error` handler flips a signal; the view renders the
  tile instead of the `img`. Shared as a small component (e.g.
  `IconOrLetter { spec: Option<String>, title: String }` in `components.rs`).
- No server change. For llamacpp specifically the user may point the config
  at `di:ollama` or any custom URL, or keep the tile.

## 4. Bookmarks absorb the yomu integration

### Bookmark native launch

- `Bookmark` (`chaos-domain/src/dashboard.rs`) gains
  `android_package: Option<String>` (serde default, skip-if-none).
- Click behavior (UI):
  - Android + package set: launch the native app via the existing
    `open_app_native` plumbing (currently in `lib.rs` for `AppNavEntry`);
    fall back to opening the URL if launch fails.
  - Desktop app and webui: plain link, `target="_blank" rel="noreferrer"`
    (the Tauri webview routes external targets to the default browser).

### Remove the apps subsystem

Delete entirely:
- `AppLink` type (`chaos-domain/src/service.rs`)
- `apps` config field (`chaos-server/src/config.rs`) and whatever endpoint
  serves it
- `AppNavEntry` component and its nav slot (`chaos-ui/src/lib.rs`)
- `/apps/:id` route and `pages/app.rs`
- Mentions in `docs/ROADMAP.md`, `docs/HANDOFF.md`, `chaos.example.toml`

Config-breaking for zeus (`[[apps]]` becomes unknown); called out in release
notes. Figment tolerates unknown keys, so an un-migrated config must not
crash the server — verify.

### Example config

`chaos.example.toml` bookmarks gain a group with:
- yomu: `url = "http://zeus:4700"`, `android_package = "xyz.tdbm.yomu"`
- Home Assistant: its zeus URL

## 5. Navbar refactor + About page

### Mobile (≤40rem)

- Replace the horizontally-scrolling bottom bar with a **fixed five-tab
  bottom bar** (yomu's `tabbar` pattern): ▦ Dash · ⛓ Links · ☀ Weather ·
  ⌂ Home · ≡ More. Icon (Unicode glyph, ~1.1rem) above a small label;
  `env(safe-area-inset-bottom)` padding; `main` gets bottom padding.
  Glyphs are placeholders — tweak freely at implementation.
- **More page** (route `/more`, reachable only from the tab): links to
  ▣ Calendar, ⚙ Settings, ⓘ About, plus account name and Logout.

### Desktop

- Sidebar keeps all entries (Calendar in place, no More) and gains the same
  glyph before each label. Sidebar foot gains an About link beside Settings.

### About page (route `/about`)

- Shows web version (`env!("CARGO_PKG_VERSION")` compiled into chaos-ui/web)
  and server version (from `/health`, already exposed as
  `HealthResponse.version`).
- Explicit sync line: "in sync ✓" when equal, "version mismatch ⚠" when not,
  and a pending state while `/health` loads / on error ("server unreachable").

## 6. Zoom & horizontal-scroll lockdown

- `crates/chaos-web/index.html` viewport meta becomes
  `width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no`.
  This kills page pinch-zoom (Android webview + browsers) and stops the page
  from stealing chart pinch gestures.
- Remove `.topbar { overflow-x: auto }` (superseded by the tab bar).
- Add `body { overflow-x: hidden }` as a guard.
- Audit remaining pages (calendar grid, services grid, wide cards/tables) so
  any wide content scrolls **inside its own container** (`overflow-x: auto`
  on the container), never the page. Fix what the audit finds.
