# UX Bug Fixes Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix four verified UX bugs: (1) dashboard widgets remount and snap their Collapsible lists shut on every poll, (2) the links pager strands on a ghost page after deleting the last item of the last page, (3) the collection parent `<select>` allows cycles (UI offers descendants; server cycle check races outside a transaction), (4) zero-duration calendar events never render.

**Architecture:** Rust workspace. Frontend is Leptos 0.8 CSR in `crates/chaos-ui` (pages in `src/pages/`, native `#[cfg(test)] mod tests` blocks already exist in `pages/home.rs`, `pages/weather.rs`, `components.rs`, `echarts.rs` and run with plain `cargo test -p chaos-ui`). Backend is Axum + sqlx/SQLite in `crates/chaos-server` (`Db` wraps a `SqlitePool`; tests use `Db::in_memory()` with `#[tokio::test]`, see `db.rs:689`). Shared types in `crates/chaos-domain` already derive `Clone + PartialEq`, which makes `Memo` payload-memoization possible.

**Tech Stack:** Rust, Leptos 0.8 (reactive_graph `RwSignal`/`Memo`/`LocalResource`), Axum, sqlx (SQLite), chrono, uuid, tokio (server tests).

## Task 1: Dashboard widgets snap shut and remount on every poll

**Files:**
- `crates/chaos-ui/src/pages/dashboard.rs:104-168` (ServicesWidget, 30 s poll)
- `crates/chaos-ui/src/pages/dashboard.rs:172-288` (DataWidget, 5 min poll)
- `crates/chaos-ui/src/pages/dashboard.rs:293-313` (Collapsible — owns `RwSignal::new(true)` today)
- `crates/chaos-ui/src/pages/dashboard.rs:317-379` (SystemdWidget — polls every 30 s; checked: it renders `systemd_rows` **without** a Collapsible, so it has no collapse-state bug, but it has the same remount-on-identical-data pattern, so it gets the memo fix only)

**Root cause:** each poll tick re-runs `services.get()` / `data.get()` inside the render closure, which tears down and rebuilds the whole widget subtree even when the payload is identical — constructing a fresh `Collapsible` (and with it a fresh `RwSignal::new(true)`, re-collapsing a user-expanded list) and recreating every `<img>`/`IconOrLetter`.

**Fix (two layers, both minimal):**
1. Hoist the `collapsed` signal out of `Collapsible` into the widget component body (created once per component instance), so even when the subtree does rebuild the collapse state survives.
2. Wrap each `LocalResource` in a `Memo` keyed on the payload (`Result<_, String>` so the error side is `PartialEq` too — `chaos_client::Error` is not `PartialEq`). `Memo` only notifies dependents when the value actually changes, so a poll returning identical data no longer rebuilds the subtree at all (fixes the `<img>` / `IconOrLetter` churn).

## Task 2: Links pagination strands on ghost page

**Files:**
- `crates/chaos-ui/src/pages/links.rs:107-136` (pager inside the `Some(Ok(page))` arm)
- `crates/chaos-ui/src/pages/links.rs` end of file (new `#[cfg(test)] mod tests`)

**Root cause:** deleting the last item on the last page bumps `refresh`; the refetch for the now-out-of-range `page_index` returns an empty page, but nothing moves `page_index` back — the user sees "Nothing here yet" and a "2 / 1" pager whose Prev button still works but shouldn't be needed.

**Fix:** extract a pure `clamp_page` helper (unit-tested natively) and, when the fetched page reveals `page_index` is past the end, clamp it — the signal write re-runs the `links` resource, which refetches the last valid page.

### Steps

- [ ] **2.1 Write the failing test first.** Append to the end of `crates/chaos-ui/src/pages/links.rs`:

      #[test]

          // 51 links at 50/page → pages 0 and 1; index 1 is valid.

          // Deleting the 51st link leaves one page; index 1 is now a ghost.

  /// Highest valid page index for `total` items at `page_size` per page —
  /// deleting the last items of the last page must not strand the pager on
  /// a ghost page ("2 / 1", empty list).
  fn clamp_page(page: u32, total: u64, page_size: u32) -> u32 {
      let pages = total.div_ceil(page_size as u64).max(1) as u32;
      page.min(pages - 1)
  }

  ```rust
  /// Reject a `parent_id` that would make `id` an ancestor of itself. Runs on
  /// the caller's transaction so the walk and the subsequent UPDATE are atomic.
  async fn ensure_no_collection_cycle(
      conn: &mut sqlx::SqliteConnection,
      id: Uuid,
      new_parent: Uuid,
  ) -> Result<()> {
      let mut cursor = Some(new_parent.to_string());
      let id = id.to_string();
      // Bounded walk: deeper nesting than this is certainly a data bug.
      for _ in 0..64 {
          let Some(current) = cursor else {
              return Ok(());
          };
          if current == id {
              return Err(DbError::Constraint(
                  "collection cannot be its own ancestor".into(),
              ));
          }
          cursor = sqlx::query_scalar::<_, Option<String>>(
              "SELECT parent_id FROM collections WHERE id = ?",
          )
          .bind(current)
          .fetch_optional(&mut *conn)
          .await?
          .flatten();
      }
      Err(DbError::Constraint("collection nesting too deep".into()))
  }
  ```

  /// `root` plus every collection nested under it (breadth-first) — none of
  /// these may become `root`'s parent without creating a cycle. Robust to
  /// pre-existing cycle data via the visited check.
  fn descendant_ids(collections: &[Collection], root: Uuid) -> Vec<Uuid> {
      let mut out = vec![root];
      let mut i = 0;
      while i < out.len() {
          let parent = out[i];
          out.extend(
              collections
                  .iter()
                  .filter(|c| c.parent_id == Some(parent) && !out.contains(&c.id))
                  .map(|c| c.id),
          );
          i += 1;
      }
      out
  }
  ```

  /// `with_depth` can never render (the subtree vanishes from the sidebar).
  #[component]
  fn CollectionOptions(
      collections: CollectionsResource,
      current: Option<Uuid>,
      exclude: Option<Uuid>,
  ) -> impl IntoView {
      move || match collections.get() {
          Some(Ok(list)) => {
              let excluded = exclude
                  .map(|id| descendant_ids(&list, id))
                  .unwrap_or_default();
              with_depth(&list)
                  .into_iter()
                  .filter(|(_, c)| !excluded.contains(&c.id))
                  .map(|(depth, c)| {
                      let label = format!("{}{}", "\u{a0}\u{a0}".repeat(depth), c.name);
                      view! {
                          <option value=c.id.to_string() selected=current == Some(c.id)>
                              {label}
                          </option>
                      }
                  })
                  .collect_view()
                  .into_any()
          }
          _ => ().into_any(),
      }
  }
  ```

  The `EditCollectionModal` call site (links.rs:805-809, `exclude=Some(id)`) needs no change — the widened semantics do the work. `EditLinkModal` passes `exclude=None` (unchanged behavior).

## Task 4: Zero-duration events never render

**Files:**
- `crates/chaos-ui/src/pages/calendar.rs:300-315` (`covers` — `end = ends_at - 1s` makes the range empty when `ends_at == starts_at`)
- `crates/chaos-ui/src/pages/calendar.rs` end of file (new `#[cfg(test)] mod tests` — the file has none today; `covers` is pure chrono date logic, so it is natively testable)

**Root cause:** the exclusive-end adjustment `ends_at - Duration::seconds(1)` is correct for real ranges (a one-day all-day event ending at next midnight must not spill into the next day) but produces `end < start` for zero-duration events, so `start <= day && day <= end` is never true and the event is invisible.
